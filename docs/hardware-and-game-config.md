# 硬件与游戏配置

## 1. 运行平台

优先平台：

- 大电视：黑客松和课堂展示主场景。
- iPad：便携演示和小范围测试。
- 电视机自带安卓浏览器：尽量支持，作为低门槛现场运行方式。

设计要求：

- 大屏上远距离能看清船号、航迹、风向、水流和规则提示。
- UI 文字不能太小，关键提示要在 2-3 秒内被观众理解。
- 操作不依赖键盘方向键，键盘只作为开发和备用输入。
- 尽量减少高性能网页特性依赖，优先保证安卓电视浏览器能打开和运行。

技术约束：

- 不强依赖 WebGPU。
- 避免大型视频背景和重度滤镜。
- 控制粒子、箭头和航迹数量。
- 图片资源压缩，首屏加载尽量轻。
- 如果电视浏览器无法稳定运行，准备外接电脑或平板浏览器作为兜底。

## 2. 蓝牙帆船模型手柄

项目的特色硬件是特制蓝牙手柄。手柄外观是一个帆船模型，玩家拨动模型上的舵杆来控制游戏里的船。

### 2.1 通道设计

系统有 4 个蓝牙连接器，每个连接器代表一条船。每个连接器只有一个轴，用来控制对应帆船的船舵。

| 蓝牙连接器 | 默认船只 | 屏幕标识 |
| --- | --- | --- |
| 连接器 1 | 1 号船 | Boat 1 |
| 连接器 2 | 2 号船 | Boat 2 |
| 连接器 3 | 3 号船 | Boat 3 |
| 连接器 4 | 4 号船 | Boat 4 |

游戏可以配置为：

- 1 条船：教学或单人计时。
- 2 条船：最适合规则演示。
- 3 条船：更接近真实拥挤场景。
- 4 条船：黑客松大电视 Demo 主模式。

### 2.2 输入含义

每个连接器输出一个舵杆轴值：

```text
rudder = -100 ~ 100
```

含义：

- `-100`：最大左舵。
- `0`：舵居中。
- `100`：最大右舵。

游戏内部建议归一化为：

```text
turn = rudder / 100
```

黑客松 Demo 不需要玩家手动调帆。帆角由系统自动处理，玩家只负责控制方向。

### 2.3 备用输入

开发和现场兜底仍保留键盘：

- 玩家 1：`A/D` 或方向键左右转舵。
- 玩家 2-4：可配置按键或自动演示输入。
- 公共：`Space` 暂停，`R` 重开。

键盘不是正式 Demo 的主要操作方式。

### 2.4 模拟控制

没有真实蓝牙手柄时，系统必须能模拟手柄：

- 键盘左右键或 `A/D` 映射为舵杆 `-100 ~ 100`。
- 自动演示脚本可以控制一条或多条船。
- 讲解模式可以用按钮或滑块控制船头角度、风向和水流。
- 项目介绍页可以让小船按脚本自动移动。

模拟控制是开发和排练的必要能力，不能依赖硬件全部到位后才调试。

## 3. 船只配置

每条船需要有独立配置：

```text
boatId
displayName
controllerChannel
color
sailNumber
enabled
startPosition
```

基本要求：

- 支持 1-4 条船开关。
- 支持通道重新绑定。
- 支持每条船的颜色和编号。
- 大屏上不能只靠颜色区分船，还需要编号、轮廓或船帆图案。

## 4. 比赛场地

场地数量不需要很多，Demo 阶段建议固定 4 个。

### 4.1 简单场地

用途：教学和第一次试玩。

结构：

```text
起航线 -> 1 标 -> 终点线
```

特点：

- 只有一个核心航标。
- 适合讲解风向、禁航角、绕标和冲线。
- 不适合展示复杂规则，但很适合新玩家上手。

### 4.2 复杂场地

用途：4 船对抗和黑客松主比赛。

结构：

```text
起航线 -> 1 标 -> 2 标 -> 3 标 -> 4 标 -> 终点线
```

特点：

- 有足够多会船和绕标机会。
- 能展示风区、水流、Layline 和路线选择。
- 适合让观众看到“战术选择导致差距”。

### 4.3 IO 场地

用途：介于教学和正式比赛之间的标准演示。

结构：

```text
起航线 -> 1 标 -> 2 标 -> 终点线
```

特点：

- 比简单场地多一次航线变化。
- 比复杂场地更容易在短时间内讲清楚。
- 适合 2-4 船规则演示。

### 4.4 规则情景场地

用途：专门演示规则。

结构：

```text
两船或四船从预设位置开始 -> 触发会船 / 换舷 / 标位场景
```

特点：

- 不追求完整比赛。
- 重点是让规则提示稳定出现。
- 可用于 Demo 现场“看，这里左舷船要让右舷船”的讲解。

## 5. 环境因素

### 5.1 风向

风向会左右摆动：

```text
windDirection = baseDirection + oscillation
```

Demo 参数建议：

- 摆动幅度：正负 8-15 度。
- 摆动周期：20-60 秒。
- 大屏上显示当前风向箭头和趋势。

风摆类型：

- 持续风摆：风向持续左摆或右摆一段时间，适合演示航线优势变化。
- 钟摆式风摆：风向围绕基准方向周期性摆动，适合演示起航和换舷时机。

### 5.2 风速

风速影响不同风角下的船速。

Demo 参数建议：

- 基准风速：8-14 节。
- 阵风区：增加 15%-35% 船速潜力。
- 弱风区：降低 10%-25% 船速潜力。

阵风应作为小面积区域处理：

- 可以移动。
- 可以增强风速。
- 可以减弱风速。
- 可以改变局部风向。
- 边缘可做渐变，避免船进出区域时速度突变过强。

### 5.3 固定水流

水流不是 MVP 核心必做项，可以交给学生作为扩展任务。若负责人先实现，只做固定水流区域，对船的对地速度产生矢量影响：

```text
对地速度 = 相对水速度 + 水流速度
```

Demo 重点：

- 水流边界要清晰可见，观众能看见箭头。
- 水流要实际改变航迹，尤其在绕标和冲线附近。
- 水流不直接改变船相对水的速度，只改变对地漂移。

### 5.4 风区

风区是局部环境变化区域：

- 进入强风区后风速变大。
- 进入风向变化区后风向左摆或右摆。
- 风区可以固定，也可以缓慢移动。

Demo 中风区应明显可见，但不要遮挡船和航标。

风区和阵风的区别：

- 风区：更稳定，可作为地图的一部分。
- 阵风：更临时、更小、更像一股一股移动的风。

## 6. 配置文件建议

后续实现时，建议把比赛配置做成数据，而不是写死在代码里：

```text
courses/
  simple.json
  complex-four-mark.json
  io-course.json
  rules-scenario.json

controllers/
  bluetooth-sail-controller.json

environment-presets/
  steady-wind.json
  shifting-wind.json
  current-demo.json
  gust-zone-demo.json
```

这样学生可以改参数和场地，而不必深入核心游戏循环。
