# 实施路线图

本文档把 PRD 拆成可执行阶段，并对照当前代码现状说明下一步该怎么推进。当前项目优先目标是黑客松 Demo 脚手架，而不是一次性完成完整训练系统。

实际团队模式是：负责人先把基础代码、页面脚手架、共享组件和主要 Demo 链路搭好；学生在两天多的有效时间里主要通过 Vibe Coding 和 AI 辅助补充规则细节、讲解内容、演示脚本和 Pitch。

## 1. 当前技术现状

当前项目是 Vite + React + PixiJS 游戏原型，核心目录如下：

- `src/app`：应用入口和游戏主界面。
- `src/components/hud`：风况、水流、倒计时、玩家状态和教练控制面板。
- `src/game/loop`：游戏循环、键盘控制、手柄控制。
- `src/game/rendering`：PixiJS 画面层，包括船、赛道、水面、风、水流和战术覆盖层。
- `src/game/systems`：船体物理、极曲线、风、水流、风区和比赛进度。
- `src/store`：Zustand 游戏状态。

已具备：

- 双船本地操控。
- 起航倒计时。
- 简化船速和转向。
- 全局风、局部风区和已有水流雏形。
- 单个上风标和终点线。
- 风、流、航迹、Layline、禁航角图层。
- 部分单元测试。

主要缺口：

- 还没有真实舵角、帆角、换舷状态和 STW/SOG 分离。
- 还没有 1-4 船配置和蓝牙帆船模型手柄通道映射。
- 起航线合法性、航段顺序和绕标方向仍很简化。
- 没有碰撞、基础航权、犯规减速和大屏规则提示。
- 没有项目介绍页、Demo 教学流程和大屏演示包装。
- 没有回放、比赛复盘和完整教学关卡。

## 2. 阶段零：黑客松 Demo 主链路

目标：由负责人先跑通大电视演示闭环，让学生后续只需要在清晰边界内补内容和微调。

范围：

- 项目介绍页：能在网页里展示项目背景、硬件手柄和核心亮点。
- Demo 流程：介绍 -> 手柄教学 -> 规则教学 -> 比赛。
- 1-4 船配置：支持启用 1、2、3、4 条船。
- 手柄映射：通道 1-4 对应船 1-4，舵杆值映射为转向比例。
- 备用输入：键盘和自动演示模式。
- 场地切换：简单场地、IO 场地、复杂 4 标场地、规则情景场地。
- 规则 10 Demo：左舷受风船撞右舷受风船时，显示规则提示并减速 3 秒。
- 大屏 UI：船号、规则提示、风区、可选水流和当前排名清晰可见。

验收：

- 打开网页后能从项目介绍进入教学，再进入比赛。
- 4 个手柄通道能控制 4 条船，或在无手柄时用备用输入演示。
- 两船交叉碰撞能稳定触发规则 10 提示。
- 犯规船能明显减速 3 秒。
- 5-7 分钟 Demo 可以完整排练。

负责人完成阶段零后，学生再进入项目。否则学生会把有限时间花在理解工程结构和修脚手架上，反而影响 Pitch。

## 3. 阶段一：基础帆船模拟

目标：让“开船”本身具备可信的帆船特征。

范围：

- 船体运动状态扩展：舵角、帆角、角速度、相对水速度、对地速度。
- 舵效模型：速度越高舵效越明显，低速转向困难，舵角过大增加阻力。
- 帆角模型：输入调帆，计算理想帆角和调帆效率。
- 禁航角：顶风减速、帆抖动和舵效下降。
- 换舷状态：迎风换舷和顺风换舷产生速度损失。
- HUD 显示：TWA、STW、SOG、调帆效率、左/右舷受风。

验收：

- 船进入禁航角后明显减速。
- 正确调帆比错误调帆更快。
- 如果启用水流，水流只影响对地航迹，不直接改变相对水速。
- 相同输入序列在相同初始条件下结果确定。

## 4. 阶段二：完整比赛闭环

目标：让比赛从起航到冲线形成可玩的完整闭环。

范围：

- 赛道数据结构：起航线、航标列表、绕标方向、终点线、航段。
- 起航合法性：起航前越线 OCS，返回后可重新起航。
- 航标顺序：按指定顺序和指定一侧绕标。
- 标门：下风门左右标可选。
- 终点判定：只允许完成全部航段且无未执行处罚的船有效冲线。
- 比赛设置：初级短航线和标准迎尾风航线。
- 结果页：名次、时间、航程、换舷次数、基础统计。

验收：

- 起航前越线不会被判合法起航。
- 错误绕标或漏标不能有效完赛。
- 两名玩家可以完成一局 3-8 分钟对抗。
- 暂停、慢放和重开不破坏比赛状态。

## 5. 阶段三：规则裁判

目标：实现可解释的主要会船规则、标位规则和处罚。

范围：

- 船体碰撞轮廓和航标碰撞轮廓。
- 左/右舷受风判定。
- 同舷重叠、前后船、上风/下风判定。
- 规则 10 的稳定 Demo 判定。
- 规则 11-13 的基础状态机。
- 规则 14-16 的简化限制提示。
- 三船长标位区和进入时重叠关系锁定。
- Demo 处罚：犯规船强制减速 3 秒。
- 训练版处罚：碰标一圈、会船犯规两圈、处罚转圈识别。
- 规则事件队列和自然语言解释。

验收：

- 左舷受风船撞击右舷受风船时能正确判断。
- 同舷重叠时能识别上风船和下风船。
- 换舷中的船被识别为避让船。
- 航权船突然转向、不给避让空间时，不会错误处罚对方。
- 标位区进入时的重叠关系被锁定。
- Demo 版本犯规可触发 3 秒减速。
- 训练版本碰标和会船犯规可分别触发一圈、两圈处罚。

## 6. 阶段四：教学与复盘

目标：把游戏从“能判罚”升级为“能训练”。

范围：

- 教学关卡系统。
- 规则情景训练加载器。
- 比赛回放数据记录。
- 关键事件索引：抢航、碰标、碰撞、航权变化、处罚开始/完成。
- 回放控制：暂停、0.25x、0.5x、1x、2x、跳转事件。
- 战术统计：Layline 误差、换舷次数、调帆效率、强风区停留时间。
- 赛后建议：最多三条高价值建议。

验收：

- 新玩家 3 分钟教学后能独立完成比赛。
- 每次判罚都显示谁犯规、违反什么规则、为什么。
- 回放中规则事件与实时比赛一致。
- 结果页不只显示胜负，还能解释关键损失。

## 7. 阶段五：扩展

目标：增强长期价值和训练场景。

可选范围：

- AI 对手。
- 在线双人。
- 教练自定义局面。
- 多船竞赛。
- 历史比赛局面复现。
- 高级船型参数和更细极曲线。

## 8. 推荐优先级

建议近期按以下顺序推进：

1. 负责人先做 Demo 主链路：项目介绍、讲解模式、比赛入口、1-4 船配置和手柄映射。
2. 负责人再做规则 10 演示：左右舷受风、碰撞、规则提示和 3 秒减速。
3. 负责人搭好共享组件边界：船、风、可选水流、场地、规则提示、事件日志。
4. 学生进入后，A 组调环境、场地和战术讲解，B 组调规则讲解、项目介绍和 Pitch。
5. 再调比赛观感：风区、阵风、风摆、场地切换、船号、排名和大屏 UI。
6. 如果时间允许，再重构船体状态和运动模型，补齐舵角、帆角、TWA、STW、SOG。
7. 最后才扩展规则 11-16、标位、碰标、回放和赛后分析。

原因是黑客松成败取决于现场演示闭环。复杂判罚和高精度物理可以逐步增强，但项目介绍、手柄控制、规则 10 提示和 4 船大屏比赛必须最先稳定。

## 9. 学生参与边界

学生实际开发时间只有两天多，并且主要用于 Pitch 训练，所以代码任务要控制在低冲突区域：

- A 组：场地配置、环境参数、战术提示、讲解文案。
- B 组：规则解释、犯规文案、项目介绍页、Demo 脚本。

学生不优先承担：

- 核心游戏循环重构。
- 复杂状态管理。
- 多人同步。
- 高精度物理。
- 完整规则引擎。

他们可以通过 Vibe Coding 与 AI 沟通，把“帆船规则和讲解逻辑”转化成文案、配置、小演示和局部 UI，而不是从零写系统。
