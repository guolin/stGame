# 共享模拟组件架构

比赛、讲解模式和项目介绍不能各写一套逻辑。这个项目的核心应是一套共享底层模拟组件，上层页面只负责“怎么展示、怎么引导、怎么编排”。

## 1. 核心原则

同一套规则在三个场景里复用：

- 比赛：玩家真实操作，组件负责模拟和判定。
- 讲解：锁定某些参数，让观众看懂船、风、水流、规则之间的关系。
- 项目介绍：用更短、更漂亮的动画展示项目亮点。

不要出现：

- 比赛里一套船速算法，讲解里另一套船速算法。
- 比赛里一套规则提示，介绍页里另一套规则提示。
- 比赛里风箭头和讲解里的风箭头视觉不一致。

## 2. 分层架构

建议拆成四层：

```text
页面编排层
  比赛页 / 讲解页 / 项目介绍页 / 赛后页

交互控制层
  手柄输入 / 键盘输入 / 自动演示输入 / 教学脚本

共享模拟层
  船体物理 / 风场 / 可选水流 / 场地 / 规则裁判 / 事件系统

共享渲染层
  船 / 帆 / 风箭头 / 水流箭头 / 航标 / 航迹 / HUD 组件
```

页面编排层可以不同，但下面三层必须尽量共用。

## 3. 共享模拟组件

### 3.1 船只组件

负责：

- 船的位置。
- 船头方向。
- 舵角。
- 自动帆角。
- 船速。
- 当前受风舷。
- 航迹。
- 犯规减速状态。

输出：

```text
BoatState
BoatVisualState
BoatTelemetry
```

三个场景的使用方式：

| 场景 | 使用方式 |
| --- | --- |
| 比赛 | 由手柄或键盘控制 |
| 讲解 | 由滑块、按钮或预设脚本控制 |
| 项目介绍 | 由自动动画脚本控制 |

### 3.2 船与风关系组件

负责：

- 根据船型读取 Polar Data。
- 计算船头与风的夹角。
- 判断是否顶风。
- 根据风角计算目标船速。
- 根据自动帆角计算效率。
- 输出“为什么快 / 为什么慢”的解释数据。

示例输出：

```text
trueWindAngle: 42
speedPotential: 0.82
sailEfficiency: 0.74
state: "closeHauled"
explanation: "迎风角度合适，但帆稍微过松"
```

讲解模式可以直接显示这些解释数据；比赛模式只显示简化状态。

Polar Data 的基本接口：

```text
getPolarSpeed(boatType, trueWindSpeed, trueWindAngle) -> targetSTW
```

水流不进入 Polar Data，只在船体物理速度组件中影响对地速度。MVP 可以先不启用水流，或只启用固定、边界清晰的水流区。

### 3.3 船体物理速度组件

负责：

- 根据目标船速平滑加速 / 减速。
- 根据舵角和速度计算转向。
- 根据可选水流计算对地速度。
- 应用犯规减速。

核心公式：

```text
对地速度 = 相对水速度 + 水流速度
最终速度 = 船速模型 * 处罚系数
```

Demo 版处罚系数：

```text
正常 = 1.0
犯规减速 = 0.35 ~ 0.5
```

### 3.4 风向标记组件

负责：

- 全局风向箭头。
- 局部风区箭头。
- 风速强弱视觉。
- 风向摆动趋势。
- 持续风摆。
- 钟摆式风摆。
- 移动阵风。

在不同页面复用：

- 比赛 HUD 左上角显示全局风。
- 赛场中显示风区。
- 讲解页显示可拖动或可观察的风箭头。
- 项目介绍页用风箭头做动态背景。

风场组件需要输出任意位置的局部风：

```text
getLocalWind(position, time) -> { directionDeg, speedKnots }
```

局部风由基准风、风摆、阵风和风区叠加得到。

### 3.5 可选水流标记组件

负责：

- 水流区域。
- 水流方向箭头。
- 水流强度。
- 船经过水流后的航迹偏移可视化。

水流不是 MVP 核心必做项。若实现，先做固定区域和清晰边界，方便学生扩展。

讲解模式中应提供开关：

```text
关闭水流 -> 船沿原航线走
打开水流 -> 船被推偏
```

### 3.6 场地组件

负责：

- 起航线。
- 航标。
- 终点线。
- 航段顺序。
- 绕标方向。
- 小地图预览。

比赛设置、比赛页和项目介绍都使用同一份场地数据。

场地示例：

```text
A 简单迎风场地
B IO 场地
C 三角标场地
D 复杂 4 标场地
```

### 3.7 规则裁判组件

负责：

- 判断左舷 / 右舷受风。
- 判断两船关系。
- 检测碰撞。
- 输出规则提示。
- 应用犯规减速事件。

输出事件：

```text
RULE_WARNING
RULE_BREACH
SLOWDOWN_STARTED
SLOWDOWN_COMPLETED
```

比赛中自动运行；讲解模式中可以锁定局面并手动触发“判定”。

### 3.8 事件系统

负责记录：

- 起航。
- 绕标。
- 进入风区。
- 进入水流区。
- 犯规。
- 减速开始 / 结束。
- 完赛。

比赛页用于事件日志；赛后页用于总结；项目介绍页可以用少量事件做演示动画。

## 4. 共享渲染组件

### 4.1 船样式

船必须表达：

- 船号。
- 船体方向。
- 帆的位置。
- 玩家颜色。
- 是否犯规减速。
- 是否被规则高亮。

建议船体不要只用颜色区分，因为大电视和投影可能色差明显。

### 4.2 风视觉

风视觉包含：

- 大方向箭头。
- 局部网格箭头。
- 强风区高亮。
- 风向摆动动画。

### 4.3 水流视觉

水流视觉包含：

- 半透明区域。
- 流向箭头。
- 强度差异。
- 航迹被推偏的对比提示。

### 4.4 规则提示视觉

规则提示视觉包含：

- 风险提示。
- 犯规提示。
- 犯规船高亮。
- 被影响船高亮。
- 减速倒计时。

比赛、讲解、项目介绍里都应该使用同一套提示组件，只是尺寸和文案不同。

## 5. 页面如何复用组件

### 5.1 比赛页

使用：

- 船只组件。
- 手柄输入。
- 船与风关系组件。
- 船体物理速度组件。
- 风向标记组件。
- 可选水流标记组件。
- 场地组件。
- 规则裁判组件。
- 事件系统。

特点：

- 全部实时运行。
- 玩家输入优先。
- 规则自动判定。

### 5.2 讲解模式

使用：

- 同一套船、风、水流、规则组件。
- 但输入来自教学脚本、滑块或预设按钮。

特点：

- 可以暂停。
- 可以慢放。
- 可以锁定局面。
- 可以显示更多解释文字。

### 5.3 项目介绍

使用：

- 船渲染。
- 风渲染。
- 水流渲染。
- 简化规则提示。
- 自动演示脚本。

特点：

- 更像 PPT。
- 每页只展示一个亮点。
- 不让观众操作复杂参数。

## 6. 推荐开发顺序

1. 先把船、风、水流、场地、规则提示都做成可复用组件。
2. 再做比赛页，验证组件能真实跑起来。
3. 再做讲解页，用同一组件锁定参数做教学。
4. 最后做项目介绍页，用同一组件包装成 PPT 式动画。

如果先做三个页面，再抽组件，最后一定会重复实现，而且规则和物理表现容易不一致。

## 7. 学生分工建议

A 组可以负责：

- 船只组件。
- 船与风关系组件。
- 船体物理速度组件。
- 风向和水流标记组件。
- 场地组件。

B 组可以负责：

- 规则裁判组件。
- 规则提示视觉。
- 事件系统。
- 讲解模式。
- 项目介绍页面。

两组共同约定数据结构：

```text
BoatState
WindState
CurrentState
CourseState
RuleEvent
RaceEvent
```

只要数据结构一致，两个小组可以并行工作。
