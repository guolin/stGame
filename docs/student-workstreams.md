# 学生分工与交付计划

项目有 4 个学生，建议分成两个两人小组。基础代码、页面脚手架、核心组件和主要工程结构最好由负责人提前搭好；学生主要在这个框架上用 Vibe Coding 和 AI 协作，把规则细节、讲解内容、演示脚本和 Pitch 打磨清楚。

这不是一个让学生从零开发完整系统的任务。更合理的目标是：

- 他们能说清楚帆船比赛是怎么回事。
- 他们能说清楚程序如何把风、水流、船和规则可视化。
- 他们能通过 AI 辅助补充规则细节、提示文案和演示场景。
- 他们能完成一次稳定、有逻辑、有互动的 Pitch。

代码分工上，A 组和 B 组要尽量减少交集，合并时很少产生冲突。

开发过程采用持续迭代和持续集成：

- 每个小组可以独立开发自己的内容，不依赖另一个小组阻塞。
- 每天至少集成一次；如果时间紧，每半天集成一次。
- 集成时先保证主 Demo 能跑，再合并新增内容。
- 不允许两天后才第一次合并，风险太高。

## 1. 推荐分组

### A 组：帆船比赛与环境组

人数：2 人

负责：

- 基于现有脚手架调参数，而不是重写底层物理。
- 梳理船的操控手感该如何向观众解释。
- 梳理舵杆输入到船体转向的映射逻辑。
- 梳理船速与风角的关系。
- 设计风向摆动、风速、风区和水流的演示说法。
- 设计 Polar Data、禁航角、阵风、持续风摆和钟摆式风摆的教学说法。
- 选择最适合 Demo 的场地和环境预设。

核心目标：

> 让观众相信这不是普通开船小游戏，而是在模拟帆船比赛里的风、水流和航线选择。

### B 组：规则裁判与 Demo 体验组

人数：2 人

负责：

- 用 AI 辅助整理规则 10-16 的 Demo 解释。
- 梳理左舷 / 右舷受风、上风 / 下风、前后船、换舷状态怎么讲。
- 设计犯规提示文案。
- 设计“犯规后减速 3 秒”的教学解释。
- 打磨项目介绍 PPT、教学流程和 Demo 脚本。
- 准备现场 Pitch 和问答。

核心目标：

> 让观众看懂为什么犯规，并且让评委理解这个项目的教学价值和技术实现思路。

## 2. 为什么不拆成三个组

理论上可以拆成：

- 帆船比赛组。
- 规则组。
- PPT 演示组。

但只有 4 个学生时，不建议拆成三个组。PPT 和演示不是独立装饰，它必须和规则提示、教学流程、比赛流程连在一起。因此建议把“规则”和“演示”放在同一组，避免最后出现功能能跑但讲不清楚的问题。

如果基础脚手架已经由负责人完成，学生的代码任务应进一步收敛：每组只改自己负责的配置、文案、演示脚本或独立页面，尽量不同时修改核心游戏循环。

## 3. 任务拆分

### 3.1 A 组任务

P0：

- 选定主 Demo 场地：简单场地、IO 场地或复杂 4 标场地。
- 选定主 Demo 天气：小风天、正常风或大风天。
- 调整风区、水流和风向摆动参数，让大屏上容易看懂。
- 设计一个持续风摆教学 Demo。
- 设计一个钟摆式风摆教学 Demo。
- 设计一个移动阵风教学 Demo。
- 设计一个固定水流区域或水流扩展示例，作为可选加分项。
- 写清楚“为什么这条船快 / 慢”的讲解词。
- 准备 1-2 个比赛中能稳定展示的战术点。

P1：

- 在 AI 帮助下优化船速、风区、水流的提示文案。
- 为比赛页面准备大屏讲解标注。
- 给每个场地写一句“适合演示什么”。

验收方式：

- 能用 60 秒讲清楚船为什么不能顶风。
- 能用 60 秒讲清楚风区如何影响比赛；水流作为可选扩展讲解。
- 能用 60 秒讲清楚 Polar Data 是“风速、风角和船速的关系”。
- 能解释持续风摆和钟摆式风摆的区别。
- 能解释主 Demo 场地为什么这样设计。
- 如果需要改代码，改动只集中在环境预设、场地配置或展示文案。

### 3.2 B 组任务

P0：

- 用自然语言写清楚规则 10 的讲解。
- 准备规则 11、12、13 的短讲解。
- 准备规则 14、15、16 的“为什么有航权也不能乱撞”的讲解。
- 设计犯规提示文案：谁犯规、违反什么规则、为什么、处罚是什么。
- 准备游戏内项目介绍页或 PPT 内容。
- 准备 Demo 流程讲稿：项目介绍 -> 讲解模式 -> 比赛。

P1：

- 用 AI 辅助生成典型两船相遇情景。
- 为每个规则场景准备“正确行为”和“错误行为”两个版本。
- 准备评委可能问到的问题和回答。
- 打磨 5-7 分钟 Pitch。

验收方式：

- 能用 90 秒讲清楚规则 10。
- 能用 2 分钟讲清楚规则 10-13 的核心区别。
- 能解释为什么 Demo 版使用“减速 3 秒”而不是完整转圈处罚。
- 能把项目介绍、规则讲解和比赛演示串成一段完整 Pitch。

## 4. 时间安排建议

总时间虽然是 4 天，但真正能用于开发的时间大约只有两天多，而且这两天主要也要服务 Pitch 训练。因此时间安排应以“能演示、能讲清楚”为中心。

### 第 0 天或开始前：负责人完成基础脚手架

负责人提前完成：

- 基础页面结构。
- 比赛入口、讲解入口、项目介绍入口。
- 1-4 船基础配置。
- 场地和环境预设。
- 基础 HUD。
- 规则 10 的最小可演示链路。
- 项目介绍页和讲解页的占位框架。

验收：

```text
打开网页 -> 项目介绍 -> 讲解模式 -> 比赛设置 -> 开始比赛 -> 触发一次规则提示
```

### 第一天：学生理解项目并补内容

- A 组：理解船、风、水流和场地，补战术讲解内容。
- B 组：理解规则 10-16，补规则讲解内容和提示文案。
- 两组一起：用自己的话讲一遍项目。

当天验收：

```text
每组能用 3 分钟讲清楚自己负责的部分，不依赖临场看稿
```

### 第二天：Vibe Coding 和 Pitch 联动

- A 组：通过 AI 调整环境预设、场地说明、战术提示。
- B 组：通过 AI 调整规则讲解、犯规文案、PPT 页面。
- 两组一起：排练完整 Demo。

当天验收：

```text
能完成一版 5-7 分钟 Pitch，并且知道每个屏幕要讲什么
```

### 第三天到第四天：主要练 Pitch 和兜底

- 减少新功能开发。
- 修复明显影响 Demo 的问题。
- 准备手柄失效、规则不触发、时间不足时的兜底讲法。
- 反复排练开场、规则讲解、比赛演示和结尾。

最终验收：

```text
项目能稳定演示，学生能说清楚项目价值、帆船知识和程序实现方式
```

## 5. 最终 Demo 优先级

最高优先级：

1. 网页能稳定打开。
2. 项目介绍能讲清楚。
3. 手柄能控制船。
4. 风、风摆、阵风和风区能被看见。
5. 规则 10 犯规能稳定演示。
6. 犯规减速 3 秒能被看见。
7. 4 人比赛能跑完或至少跑 2 分钟。

次优先级：

- 规则 11、12、13。
- 规则 14、15、16 的细化。
- 复杂赛后分析。
- 真实转圈处罚。
- 在线联机。

规则教学和风摆教学都要做。两者形式类似，都是“一个可交互局面 + 一段讲解 + 一个结果反馈”，不要把它们理解成两套完全不同的开发工作。

## 6. 合并与代码边界

为了减少冲突，建议提前划清边界：

| 负责人 | 主要改动区域 |
| --- | --- |
| 负责人 | 核心脚手架、游戏循环、基础页面、共享组件 |
| A 组 | 场地配置、环境参数、战术讲解文案 |
| B 组 | 规则讲解文案、项目介绍/PPT、Demo 脚本 |

学生尽量避免同时修改：

- 核心游戏循环。
- 船体物理核心算法。
- 全局状态结构。
- 路由和页面状态机。

如果必须改核心代码，先由负责人给出明确文件和改法，再让学生用 AI 辅助完成局部改动。

## 7. 持续集成节奏

### 7.1 集成频率

推荐节奏：

| 时间 | 集成目标 |
| --- | --- |
| 每天上午结束 | 合并上午新增内容，确认主 Demo 还能打开 |
| 每天下午结束 | 合并当天完整成果，做一次完整演示 |
| Demo 前一天 | 只修阻塞问题，不再大改结构 |

如果一天开发时间较短，至少做到：

```text
每天结束前必须集成一次，并跑一遍主 Demo
```

### 7.2 小组独立开发原则

A 组和 B 组开发时尽量不互相依赖：

- A 组可以独立调整场地、天气、风区、水流、战术讲解。
- B 组可以独立调整规则讲解、PPT、犯规文案、Demo 脚本。
- 两组通过约定好的数据结构和文档协作，不临时改对方文件。

如果某个任务需要同时改 A/B 两组内容，先拆成两个小任务：

```text
A 组：提供数据或配置
B 组：读取数据并展示
```

### 7.3 集成检查清单

每次集成后检查：

- 首页能打开。
- 项目介绍能进入。
- 讲解模式能进入。
- 比赛设置能选择人数、难度和场地。
- 至少一场 2 船比赛能开始。
- 规则 10 Demo 能触发或规则情景能播放。
- 没有明显样式错位。
- 大屏关键文字仍然可读。

### 7.4 冲突处理

如果出现冲突：

1. 先保留能跑 Demo 的版本。
2. 再逐个合入小组内容。
3. 文案冲突由负责讲解的人决定。
4. 配置冲突由负责 Demo 场地的人决定。
5. 核心代码冲突由负责人决定。

### 7.5 集成验收口径

每次集成不要求功能完整，但必须满足：

```text
主分支始终能打开、能进入 Demo、能开始一场比赛
```

这比“每个人本地都做了很多东西”更重要。
