# 持续迭代与集成流程

本项目时间短、Demo 压力大，开发方式必须是持续迭代和持续集成，而不是最后一天集中合并。

## 1. 目标

集成流程要保证：

- A 组和 B 组可以独立开发。
- 每天或每半天能把成果合到一起。
- 主 Demo 始终可运行。
- 问题尽早暴露，避免最后集中崩掉。

## 2. 分支和工作方式

推荐最简单的方式：

```text
main
  始终保持可演示

group-a
  帆船比赛与环境组

group-b
  规则裁判与 Demo 体验组
```

如果学生不熟悉 git 分支，也可以用文件边界代替分支边界：

- A 组只改场地、环境、战术文案相关文件。
- B 组只改规则、讲解、PPT、Demo 脚本相关文件。
- 负责人负责合并到主版本。

## 3. 半天集成节奏

推荐节奏：

```text
上午开发
-> 午前集成
-> 午后开发
-> 晚前集成
-> 完整 Demo 排练
```

每次集成只问三个问题：

1. 网页还能打开吗？
2. 主 Demo 还能跑吗？
3. 新增内容有没有帮助 Pitch？

如果答案是否定的，先回到上一个能演示的版本。

## 4. 每次集成检查清单

必须检查：

- 首页能打开。
- 项目介绍能进入。
- 讲解模式能进入。
- 比赛入口能进入。
- 能选择人数、难度 / 天气和场地。
- 能开始至少一场 2 船比赛。
- 风、水流、船和规则提示没有明显消失。
- 规则 10 或规则情景 Demo 能播放。
- 赛后或退出流程可用。

可选检查：

- 4 船模式。
- 蓝牙手柄。
- 安卓电视浏览器。
- 低性能模式。
- 海报、传单、网站二维码。

## 5. 集成优先级

优先级从高到低：

1. 主 Demo 可运行。
2. 比赛能开始。
3. 讲解能讲清楚。
4. 规则提示能出现。
5. 大屏视觉好看。
6. 代码结构优雅。

黑客松阶段，不能为了代码结构重构牺牲 Demo 稳定性。

## 6. 小组交付格式

每个小组每半天交付时，必须说明：

```text
我们改了什么
影响哪些页面
如何演示
有没有风险
需要负责人帮忙集成什么
```

示例：

```text
A 组：
改了 IO 场地的风区和水流参数。
影响比赛页面和讲解模式中的水流 Demo。
演示方式：选择 IO 场地 + Demo 综合环境。
风险：4 船时风箭头可能太密。
```

## 7. 冻结时间

Demo 前一天应进入冻结：

- 不再大改页面结构。
- 不再改核心物理。
- 不再新增复杂规则。
- 只修阻塞演示的问题。
- 主要时间用于排练 Pitch。

冻结后允许改：

- 文案。
- PPT 内容。
- 海报和传单。
- 明显样式问题。
- Demo 兜底按钮。

## 8. 失败兜底

如果集成后出现严重问题：

- 回到最近能演示的版本。
- 切换到 2 船模式。
- 使用规则情景 Demo 替代完整比赛。
- 使用键盘或模拟输入替代手柄。
- 使用外接浏览器设备替代电视自带浏览器。

最终目标不是功能最多，而是现场稳定、讲得清楚。
